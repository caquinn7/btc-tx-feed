# Plan: Deploy btc_tx_feed to Fly.io

## Overview

Deploy this Phoenix 1.8 + Gleam app to Fly.io using a multi-stage Dockerfile and fly.toml. The main complexity vs. a plain Gleam app: OTP release build, Gleam toolchain installation in Docker, the Gleam pre-compile ordering workaround, SQLite NIF compilation, and a persistent volume for the database.

## Files to create

- `Dockerfile` (repo root)
- `.dockerignore` (repo root)
- `fly.toml` (repo root)
- `.github/workflows/fly-deploy.yml`
- `lib/btc_tx_feed/release.ex` ŌĆö generated by `mix phx.gen.release`
- `rel/overlays/bin/server` ŌĆö generated by `mix phx.gen.release`

---

## Phase 1 ŌĆö Project prep

1. **Run `mix phx.gen.release`** to generate:
   - `lib/btc_tx_feed/release.ex` (migrate/rollback helpers for SSH console use)
   - `rel/overlays/bin/server` (sets PHX_SERVER=true before booting the release)
   This is a prerequisite for Phases 2 and 3.

2. **Create `.dockerignore`** at repo root. Exclude: `_build/`, `deps/`, `assets/node_modules/`, `priv/static/assets/`, `priv/static/cache_manifest.json`, `*.db`, `*.db-shm`, `*.db-wal`, `.git/`, `test/`.

---

## Phase 2 ŌĆö Dockerfile

3. **Create `Dockerfile`** with two stages:

   **Builder stage** (`FROM elixir:1.19-otp-28 AS builder`):
   - `apt-get install -y build-essential cmake git curl` (exqlite NIF needs gcc + cmake)
   - Download and install Gleam v1.14.0 static Linux binary from GitHub releases into `/usr/local/bin`
   - `mix archive.install hex mix_gleam "~> 0.6.2" --force`
   - `mix local.hex --force && mix local.rebar --force`
   - Set `MIX_ENV=prod`
   - Copy `mix.exs` + `mix.lock`, run `mix deps.get --only prod`
   - Pre-compile Gleam packages in order: `mix deps.compile gleam_stdlib gleam_crypto` (same ordering workaround as CI ŌĆö prod env excludes gleeunit so only two packages needed)
   - `mix deps.compile`
   - Copy `config/`, `priv/`, `lib/`, `assets/`
   - `mix assets.deploy` (tailwind minify + esbuild minify + phx.digest)
   - `mix compile`
   - `mix release`

   **Runner stage** (`FROM debian:bookworm-slim`):
   - `apt-get install -y libstdc++6 openssl ca-certificates libncurses5` (ERTS runtime deps)
   - Copy `/app/_build/prod/rel/btc_tx_feed` from builder
   - `ENV PHX_SERVER=true`
   - `CMD ["/app/bin/server"]`

---

## Phase 3 ŌĆö fly.toml

4. **Create `fly.toml`** with:
   - `app` name matching what `fly launch` assigns
   - `primary_region = "dfw"`
   - `[http_service]` ŌĆö `internal_port = 4000`, `force_https = true`
   - Health check at `/`
   - `[[vm]]` ŌĆö `size = "shared-cpu-1x"`
   - `[mounts]` ŌĆö `source = "btc_tx_feed_data"`, `destination = "/data"`

---

## Phase 4 ŌĆö CI/CD workflow

5. **Create `.github/workflows/fly-deploy.yml`**:
   - Triggered by `workflow_run` on the "Test" workflow, `branches: ["main"]`, `types: ["completed"]`
   - Job condition: `if: github.event.workflow_run.conclusion == 'success'`
   - Steps: `actions/checkout@v4`, `superfly/flyctl-actions/setup-flyctl@master`, `flyctl deploy --remote-only`
   - Requires `FLY_API_TOKEN` repo secret (set once in GitHub repo settings ŌåÆ Secrets)

---

## Phase 5 ŌĆö Fly.io provisioning

6. `fly launch --no-deploy` ŌĆö creates the app, generates base fly.toml (merge/replace with ours)
7. `fly volumes create btc_tx_feed_data --region dfw --size 1`
8. Set secrets:
   ```
   fly secrets set SECRET_KEY_BASE=$(mix phx.gen.secret)
   fly secrets set PHX_HOST=<app-name>.fly.dev
   ```
9. Add `FLY_API_TOKEN` to GitHub repo secrets (`fly tokens create deploy -x 999999h`)

---

## Phase 6 ŌĆö First deploy and migrate

10. Push to `main` ŌĆö CI runs tests, on success fly-deploy.yml triggers `flyctl deploy --remote-only`
11. Run Ecto migrations (first deploy only):
    ```
    fly ssh console -C "/app/bin/btc_tx_feed eval 'BtcTxFeed.Release.migrate()'"
    ```

---

## Verification

1. `fly status` ŌĆö confirm app is running
2. `fly logs` ŌĆö check for startup errors (SECRET_KEY_BASE missing, DB path issues, WebSocket connection to mempool.space)
3. Open `https://<app-name>.fly.dev` ŌĆö LiveView loads and live mempool txids appear
4. `fly ssh console` ŌĆö verify `/data/btc_tx_feed.db` exists and is non-empty after migrations

## Decisions / scope

- SQLite on a persistent volume (already configured in runtime.exs at `/data/btc_tx_feed.db`) ŌĆö no Postgres needed
- Using `debian:bookworm-slim` runner (not Alpine) to avoid musl/glibc friction with exqlite NIF
- Gleam binary installed via curl in builder (mirrors CI's `erlef/setup-beam` approach, avoids version mismatch from Gleam base image)
- No custom health check endpoint added ŌĆö using default `/` for now
- Single Fly machine (no clustering) ŌĆö `DNS_CLUSTER_QUERY` not set
- Migrations run manually via SSH console on first deploy (not automated in release boot)
